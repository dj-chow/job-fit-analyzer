# Software Engineering Scoring Criteria

## The Core Question
"Can this person design, build, and maintain software systems at the scale and complexity this role requires?"

## Scored Dimensions (5)

### 1. Technical Depth and Relevance (Weight: 30%)
Do their technical skills match what the role needs? This is the highest-weighted dimension because engineering hiring is fundamentally about technical capability.

| Level | What It Looks Like |
|---|---|
| Strong (80-100) | Direct experience with the role's core tech stack or closely adjacent technologies. Has worked at relevant scale. Can point to system design decisions they made. |
| Solid (60-79) | Overlapping tech stack with some gaps. Has worked on similar types of systems (backend, frontend, infra) even if specific tools differ. |
| Partial (40-59) | General software engineering experience but different tech domain. A backend engineer applying for a frontend role, for example. Skills transfer but ramp time is real. |
| Weak (20-39) | Limited overlap. Different stack, different system type, different scale. |
| Gap (0-19) | No relevant technical experience. |

Note: tech stacks evolve fast. A strong engineer can learn a new framework in weeks. Weight the TYPE of system (distributed, real-time, data-intensive, user-facing) more than specific tool names.

### 2. System Design and Architecture (Weight: 25%)
Can they think beyond individual features to design systems that work at scale?

| Level | What It Looks Like |
|---|---|
| Strong (80-100) | Designed systems handling significant load. Made architecture decisions with business and technical tradeoffs. Evidence of thinking about reliability, scalability, and maintainability. |
| Solid (60-79) | Contributed to system design. Understands architectural patterns. Has opinions about tradeoffs. |
| Partial (40-59) | Works within existing architectures. Implements features well but hasn't designed systems. |
| Weak (20-39) | Early career. Focused on individual tasks rather than system-level thinking. |
| Gap (0-19) | No system design evidence. |

This dimension matters more at senior and staff levels. For junior roles, weight it lower (15% instead of 25%) and shift weight to Dimension 1.

### 3. Shipping and Impact (Weight: 20%)
Did their code reach production? Did it matter?

| Level | What It Looks Like |
|---|---|
| Strong (80-100) | Shipped to production regularly. Can point to measurable impact (performance improvements, user-facing features, reliability gains). Led technical projects end-to-end. |
| Solid (60-79) | Ships regularly. Some evidence of impact. Contributes to team delivery. |
| Partial (40-59) | Has shipped but impact is unclear or incremental. |
| Weak (20-39) | Academic or research-focused. Limited production experience. |
| Gap (0-19) | No shipping evidence. |

### 4. Collaboration and Communication (Weight: 15%)
Can they work with other engineers, PMs, and stakeholders effectively?

| Level | What It Looks Like |
|---|---|
| Strong (80-100) | Mentors others. Drives technical discussions. Works effectively across teams. Evidence of code reviews, design docs, or technical leadership. |
| Solid (60-79) | Good team player. Communicates technical decisions clearly. |
| Partial (40-59) | Works well individually but limited evidence of cross-team collaboration. |
| Weak (20-39) | Solo contributor with minimal team interaction. |
| Gap (0-19) | No collaboration evidence. |

### 5. Engineering Practices (Weight: 10%)
Do they follow and promote good engineering practices?

| Level | What It Looks Like |
|---|---|
| Strong (80-100) | Evidence of testing discipline, CI/CD practices, code review culture, documentation habits. Has improved engineering processes for their team. |
| Solid (60-79) | Follows good practices. Some evidence of testing and process awareness. |
| Partial (40-59) | Basic awareness but limited evidence of disciplined practices. |
| Weak (20-39) | No mention of testing, reviews, or engineering process. |
| Gap (0-19) | Evidence suggests poor practices (no testing, no reviews). |

## Seniority Calibration

| Level | Scope Expectation | Technical Bar |
|---|---|---|
| Junior (0-2 years) | Tasks and small features. Needs guidance. | Can implement well-defined problems. Learning patterns and tools. |
| Mid (2-5 years) | Full features end-to-end. Minimal direction needed. | Solid in core stack. Starting to think about system design. |
| Senior (5-10 years) | Defines technical approach for ambiguous problems. Mentors others. | Deep in chosen domain. Makes architecture decisions. |
| Staff+ (8+ years) | Sets technical direction across teams. Influences org-wide decisions. | Broad and deep. Can evaluate tradeoffs across multiple systems. |

## Common Scoring Traps

- Don't equate "years" with "seniority." A 3-year engineer at a fast-moving startup may have senior-level skills. A 10-year engineer at a slow enterprise might still operate at mid-level.
- Open source contributions are a positive signal but their absence means nothing. Most strong engineers don't contribute to open source.
- FAANG experience creates a halo effect. A Google engineer isn't automatically better. But they've likely dealt with scale, which is genuinely relevant for roles that need it.
- "Full-stack" is overloaded. Check whether the role actually needs full-stack skills or if they just put it in the JD because everyone does.
