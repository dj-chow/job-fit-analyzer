# Work Activity Decomposition Framework

Evaluate candidates by what they DID, not what their title said. Decompose every role into these seven activity categories.

## The Seven Activity Categories

### 1. Strategic / Vision Work
- Setting direction and priorities for a team, product, or initiative
- Making tradeoff decisions (scope, timeline, resources, quality)
- Defining success metrics and OKRs
- Aligning stakeholders on outcomes and direction
- Market positioning, competitive strategy, go-to-market planning

### 2. Discovery / Research Work
- Identifying problems worth solving through user research, data, or stakeholder input
- Conducting user/customer interviews, surveys, usability studies
- Market and competitive analysis
- Data analysis for decision-making (not just reporting)
- Translating ambiguous situations into structured problem statements

### 3. Definition / Specification Work
- Writing requirements (PRDs, specs, user stories, acceptance criteria)
- Scoping work for engineering or other delivery teams
- Making build-vs-buy decisions
- Defining APIs, data models, system contracts
- Translating business needs into technical requirements

### 4. Execution / Delivery Work
- Managing timelines, sprints, and delivery cadence
- Coordinating across multiple teams or workstreams
- Removing blockers and managing dependencies
- Tracking progress against goals and adjusting plans
- Release management and launch coordination

### 5. Growth / Optimization Work
- Running experiments (A/B tests, feature flags, multivariate)
- Funnel analysis and conversion optimization
- Activation, engagement, retention metric ownership
- SEO, content strategy, demand generation
- Iterating on product or process based on quantitative results

### 6. Technical / Systems Work
- Architecture decisions and system design
- Data modeling, pipeline design, ETL processes
- Infrastructure and platform work
- Code review, technical debt management
- Security, compliance, and reliability engineering

### 7. Communication / Influence Work
- Cross-functional alignment without direct authority
- Executive communication (board decks, leadership updates, steering committees)
- Change management and organizational adoption
- Conflict resolution and negotiation
- Evangelizing decisions, driving buy-in across teams

## Title-to-Work Mismatch Patterns

Use this table to recognize when a candidate's title doesn't reflect their actual work. When you see these titles, look for the corresponding evidence signals.

| Official Title | Often Actually Doing | Evidence Signals to Look For |
|---|---|---|
| Solutions Architect | Product Management | Prioritized backlogs, wrote requirements/PRDs, ran discovery workshops, made build-vs-buy decisions, aligned stakeholders on roadmap |
| Business Analyst | Data Engineering | Built SQL pipelines, designed data models, wrote ETL logic, maintained data infrastructure, automated reporting |
| Project Manager | Program Management / Product Ops | Managed cross-project dependencies, owned portfolio-level reporting, established org-wide processes, drove operational improvements |
| Marketing Manager | Growth Product Management | Ran A/B experiments, owned activation/retention metrics, built growth loops, used product analytics tools, managed conversion funnels |
| Engineering Manager | Technical Product Management | Owned technical roadmap, made prioritization tradeoffs, defined technical requirements, drove architecture decisions with business context |
| Management Consultant | Strategy / Product / BizOps | Framed ambiguous problems, built business cases, ran stakeholder interviews, synthesized data into recommendations, drove implementation |
| Customer Success Manager | Product Management (voice-of-customer) | Aggregated user feedback into themes, influenced roadmap priorities, defined feature requirements from pain points, ran beta programs |
| UX Researcher | Product Strategy | Ran discovery research shaping product direction, defined user segments, validated product-market fit hypotheses |
| Scrum Master | Product Manager (in small orgs) | Owned backlog prioritization, made scope decisions, defined acceptance criteria, interfaced with business stakeholders |
| Technical Lead / Staff Engineer | Technical Product Manager | Drove architecture decisions with business rationale, owned technical strategy, scoped multi-quarter initiatives |
| Operations Manager | Program Manager / Chief of Staff | Coordinated cross-functional initiatives, built reporting systems, drove process improvements, managed executive communication |
| Data Analyst | Data Engineer (partial) | Wrote complex SQL beyond reporting, built reusable data models, automated pipelines, maintained data quality processes |
| Sales Engineer / Pre-Sales | Solutions Architecture / Product | Translated customer requirements into technical specs, influenced roadmap from field feedback, scoped custom implementations |
| Product Owner (in Scrum) | Full Product Manager | Owned product strategy beyond sprint-level decisions, conducted user research, defined product vision, managed stakeholder relationships |

## Adjacency Dimensions

When evaluating whether experience is "adjacent," consider these five dimensions:

1. **Skill adjacency** — Do the roles share 60%+ of the same core skills? (e.g., Project Manager and Program Manager share execution, stakeholder management, planning)
2. **Activity adjacency** — Are they performing the same work activities in different contexts? (e.g., Marketing Manager running experiments and Growth PM running experiments)
3. **Domain adjacency** — Different role in the same domain? (e.g., Data Analyst to Data Engineer — same data domain, different depth)
4. **Altitude adjacency** — Same skills at a different scope? (e.g., team lead to department director — same leadership skills, broader scope)
5. **Perspective adjacency** — Same problem space, different vantage point? (e.g., Engineering Manager and Technical PM both answer "what should we build" from different angles)
