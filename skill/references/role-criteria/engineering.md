# Software Engineering Scoring Criteria

## The Core Question
"Can this person design, build, and maintain software at the scale and depth this role needs?"

## Scored Dimensions (5)

### 1. Technical Depth and Fit (Weight: 30%)
Do their tech skills match what the role needs? This has the most weight because dev hiring is about tech skill at its core.

| Level | What It Looks Like |
|---|---|
| Strong (80-100) | Direct work with the role's core tech stack or close tools. Has worked at the right scale. Can point to system design calls they made. |
| Solid (60-79) | Tech stack overlaps with some gaps. Has worked on the same type of systems (backend, frontend, infra) even if tools differ. |
| Partial (40-59) | General dev work but in a different tech area. A backend dev going for a frontend role, for example. Skills transfer but ramp time is real. |
| Weak (20-39) | Little overlap. Different stack, system type, and scale. |
| Gap (0-19) | No fitting tech work. |

Note: tech stacks change fast. A strong dev can learn a new tool in weeks. Weight the TYPE of system (spread out, real-time, data-heavy, user-facing) more than tool names.

### 2. System Design and Structure (Weight: 25%)
Can they think past single features and design systems that work at scale?

| Level | What It Looks Like |
|---|---|
| Strong (80-100) | Built systems that handle big loads. Made design calls with business and tech tradeoffs. Proof of thinking about uptime, scale, and long-term care. |
| Solid (60-79) | Helped with system design. Knows common patterns. Has views on tradeoffs. |
| Partial (40-59) | Works within current setups. Builds features well but has not designed systems. |
| Weak (20-39) | Early career. Focused on single tasks, not system-level thinking. |
| Gap (0-19) | No system design proof. |

This area matters more at senior and staff levels. For junior roles, weight it lower (15% not 25%) and shift weight to Dimension 1.

### 3. Shipping and Impact (Weight: 20%)
Did their code reach production? Did it matter?

| Level | What It Looks Like |
|---|---|
| Strong (80-100) | Shipped to production often. Can point to real impact (speed gains, user features, uptime wins). Led tech projects end-to-end. |
| Solid (60-79) | Ships often. Some proof of impact. Helps the team deliver. |
| Partial (40-59) | Has shipped but impact is unclear or small. |
| Weak (20-39) | Work is academic or research-based. Little production contact. |
| Gap (0-19) | No shipping proof. |

### 4. Teamwork and Communication (Weight: 15%)
Can they work with other devs, PMs, and partners well?

| Level | What It Looks Like |
|---|---|
| Strong (80-100) | Teaches others. Drives tech talks. Works well across teams. Proof of code reviews, design docs, or tech leadership. |
| Solid (60-79) | Good team player. Shares tech calls clearly. |
| Partial (40-59) | Works well alone but little proof of cross-team work. |
| Weak (20-39) | Solo builder with little team contact. |
| Gap (0-19) | No teamwork proof. |

### 5. Dev Practices (Weight: 10%)
Do they follow and push good dev habits?

| Level | What It Looks Like |
|---|---|
| Strong (80-100) | Proof of testing habits, CI/CD use, code review culture, and docs. Has made dev processes better for their team. |
| Solid (60-79) | Follows good habits. Some proof of testing and process sense. |
| Partial (40-59) | Basic grasp but little proof of strong habits. |
| Weak (20-39) | No mention of testing, reviews, or dev process. |
| Gap (0-19) | Signs of poor habits (no testing, no reviews). |

## Seniority Calibration

| Level | Scope Expected | Technical Bar |
|---|---|---|
| Junior (0-2 years) | Tasks and small features. Needs help. | Can build well-scoped problems. Learning patterns and tools. |
| Mid (2-5 years) | Full features end-to-end. Little guidance needed. | Solid in core stack. Starting to think about system design. |
| Senior (5-10 years) | Sets the tech path for fuzzy problems. Teaches others. | Deep in their field. Makes design calls. |
| Staff+ (8+ years) | Sets tech direction across teams. Shapes org-wide calls. | Broad and deep. Can weigh tradeoffs across many systems. |

## Common Scoring Traps

- Don't equate "years" with "level." A 3-year dev at a fast startup may have senior skills. A 10-year dev at a slow firm might still work at mid-level.
- Open source work is a good sign, but its absence means nothing. Most strong devs don't do open source.
- FAANG work creates a halo effect. A Google dev is not always better. But they have likely dealt with scale, which matters for roles that need it.
- "Full-stack" is vague. Check if the role truly needs full-stack skills or if it's just a buzzword in the JD.
